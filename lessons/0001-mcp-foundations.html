1. The Problem MCP Solves

Before MCP, connecting an AI model to an external tool — a database, a calendar, a file system — meant writing custom integration code for every combination. Five AI apps × ten data sources = fifty bespoke connectors, each fragile and non-transferable.

MCP (Model Context Protocol) is an open standard that defines one protocol for that connection. Anthropic published it in late 2024; by mid-2025 it had been adopted by OpenAI, Google, and Microsoft and donated to the Linux Foundation.

The USB-C Analogy
MCP is to AI applications what USB-C is to devices. Instead of one charger per laptop brand, you get one standard port. Build an MCP server once, and any MCP-compatible host — Claude Desktop, VS Code Copilot, Cursor, your own Python agent — can use it.

Host

The host is the AI application — Claude Desktop, VS Code, or your own Python agent. The host owns the LLM interaction and decides which tools to call. It creates and manages one client per server it talks to. [source]

Client

An MCP client is a connection object inside the host, dedicated to one server. Think of it as the host's "phone line" to a specific server. One server = one client. The client handles the protocol details so the host doesn't have to.

Server

An MCP server is a program that exposes capabilities — tools, data, and templates — for the host to use. It can run locally (as a subprocess, via stdio) or remotely (over HTTP). In this workshop, you will build the server.

3. The Three Primitives

Servers expose capabilities using exactly three primitive types. Every MCP server is just a combination of these three things:

🔧
Tool
Executable

A function the LLM can call to do something: write a file, query a database, send an API request. Think of it like a POST endpoint.

📄
Resource
Read-only Data

A data source the LLM can read: a file's contents, a database schema, a feed. Addressed by a URI like notes://all. Think of it like a GET endpoint.

💬
Prompt
Template

A reusable, parameterised prompt template the host can inject into conversations — system prompts, few-shot examples, slash commands.

Mental Model
A tool changes the world. A resource describes the world. A prompt shapes how the LLM talks about the world.

4. How They Communicate

MCP uses JSON-RPC 2.0 — a simple request/response protocol over plain JSON. You don't need to write JSON-RPC yourself (FastMCP handles it), but it helps to understand the flow:

  1. Initialize: The client and server shake hands, announcing which capabilities each supports.
  2. Discover: The client calls tools/list to learn what tools exist.
  3. Execute: When the LLM wants to use a tool, the client calls tools/call with the tool name and arguments.
  4. Respond: The server runs the function and returns a result — usually a text or structured JSON payload.
// Discovery request
{ "method": "tools/list" }

// Server response
{ "tools": [{ "name": "create_note", "description": "Create a new note", ... }] }

// Execution request
{ "method": "tools/call", "params": { "name": "create_note", "arguments": { "title": "Ideas" } } }

// Result
{ "content": [{ "type": "text", "text": "Note 'Ideas' created with id 42" }] }

Transport: stdio vs HTTP

stdio (local): The host spawns the server as a subprocess. They communicate through stdin/stdout. Fast, simple, no networking. This is what we'll use in this workshop.
HTTP (remote): The server runs as a web service; clients connect over the network. Used for production, shared servers.
